# Identity

## Overview

Shepherd publishes signed Nostr events (patch-thread replies) under a reviewer-owned identity. Today that identity is configured out of band — the reviewer must place a secret key or bunker URI in an environment variable or config file before launching the app, with no in-app way to set, view, or change it. This feature adds an in-app identity screen so a reviewer can log in by entering an existing Nostr secret key (`nsec`) or a NIP-46 bunker connection URI, or generate a brand-new local identity if they do not have one, directly from the application UI — without touching the terminal or editing config files.

The configured identity persists across app launches so the reviewer does not re-enter it per session, and it is the identity under which the app's published replies are signed and attributed. A reviewer who already configured an identity out of band (via the existing environment/config paths) is not forced through the screen; the app continues to honor those sources. The screen is the friendly, self-service path for reviewers who have not set up an identity.

**Why**: Nostr identity setup is the single biggest friction point between "I installed Shepherd" and "I can publish review comments to a patch thread." Requiring a reviewer to discover `~/.config/nostr/identity`, encode a bech32 secret key, and manage it with shell tools excludes everyone who is not already a Nostr power user. An in-app login screen with a "create a new identity" option removes that barrier. Reviewers who keep their secret key in a NIP-46 remote signer (a bunker) get the same in-app convenience by pasting their bunker URI instead, so a raw secret key never has to live on the review host. The existing out-of-band path is kept for users who prefer it.

**Scope**: This covers both in-app identity forms — logging in with an existing `nsec`, creating a new local identity, and logging in with a NIP-46 bunker URI. It does not introduce account recovery, multi-account switching, or relay configuration UI. Those remain roadmap candidates. A reviewer cannot create a bunker in-app (a bunker is an external service the reviewer brings); the in-app bunker path is login-only.

## User Stories

### US-id-1: Log in with my existing nsec
**As a** reviewer who already has a Nostr identity, **I want to** paste my `nsec` into the app and log in, **so that** my published review replies are signed under my existing identity without me having to put the key in a config file or environment variable.

### US-id-2: Create a new identity when I don't have one
**As a** reviewer who is new to Nostr, **I want to** generate a fresh identity inside the app with one click, **so that** I can start publishing signed review replies immediately without first going elsewhere to create a key.

### US-id-3: Stay logged in across launches
**As a** reviewer, **I want** my logged-in identity to persist across app restarts, **so that** I do not have to re-enter my `nsec` every time I open Shepherd.

### US-id-4: See which identity I'm using
**As a** reviewer, **I want** to see a clear indication of my active identity (and be able to confirm it is the one I expect) once I have logged in, **so that** I know whose name will appear on the replies I publish before I submit them.

### US-id-5: Log out and switch identities
**As a** reviewer, **I want to** log out (forget the app-stored identity) and log in with a different `nsec`, **so that** I can rotate keys or use a different identity without editing files by hand.

### US-id-6: Keep my secret key safe
**As a** reviewer, **I want** my `nsec` to be stored securely and never written to disk in plaintext, **so that** another application or user on my machine cannot trivially read my secret key.

### US-id-7: Skip the screen if I already configured identity out of band
**As a** reviewer who already set up identity via the existing environment/config paths, **I want** the app to use that identity and not block me on the login screen, **so that** my existing setup keeps working unchanged.

### US-id-8: Log in with my bunker instead of a raw nsec
**As a** reviewer who keeps my Nostr secret key in a NIP-46 remote signer (a bunker), **I want to** paste my bunker connection URI into the app and log in, **so that** my published review replies are signed under my identity by the remote signer without my raw secret key ever being present on this machine.

## Requirements

### Core Behavior

The identity screen lets a reviewer provide or create a Nostr identity in-app, and the app remembers it so signed replies are attributed to that identity across launches. It supports two identity forms: a local secret key (`nsec`) and a NIP-46 bunker connection URI.

- **Login with nsec** `FR-id-nsec-login`: The app provides an in-app screen where the reviewer enters an existing Nostr secret key in bech32 (`nsec1...`) form. On submit, the app validates the key, derives the corresponding public key, and adopts it as the active identity for signing and attributing published replies. An invalid key (malformed bech32, wrong prefix, or not a valid 32-byte secp256k1 secret) is rejected with a clear error and does not become the active identity.
- **Create new identity** `FR-id-create-new`: On the same screen, the app offers an action to generate a brand-new Nostr identity. Generating an identity produces a fresh, cryptographically valid secret key (a random 32-byte secp256k1 secret), derives its public key, adopts it as the active identity, and displays the resulting `nsec` so the reviewer can back it up. The generated key is the reviewer's permanent identity until they log out or replace it; it is not regenerated on each launch.
- **Persist identity across launches** `FR-id-persistence`: Once a reviewer logs in or creates an identity in-app, that identity persists across app launches. The reviewer does not re-enter the `nsec` or bunker URI per session. What is stored is the minimum needed to restore the identity: the secret key for a local-key identity, or the bunker URI for a bunker identity. Stored material is held securely appropriate to the platform (never in plaintext on disk); the specific storage mechanism is an engineering decision. If the stored identity cannot be read at launch (e.g. secure storage is unavailable or corrupted), the app presents the login screen again rather than failing silently. For a bunker identity, persistence means the stored URI is used to reconnect to the remote signer on launch (see `FR-id-bunker-persist`).
- **Show created nsec for backup** `FR-id-show-new-nsec`: When a new identity is created, the app displays the generated `nsec` to the reviewer so they can copy and back it up. The app makes clear this is the only time the raw key is shown in full and that losing it means losing access to the identity. The reviewer confirms they have backed it up before the screen dismisses. This requirement does not apply to logging in with an existing `nsec` (the reviewer already has it).
- **Active identity indicator** `FR-id-active-indicator`: Once an identity is active (whether from the in-app screen or an out-of-band source), the app surfaces which identity is in use, consistent with the existing reviewer-identity indicator. The reviewer can confirm at a glance whose public key their published replies will be attributed to.
- **Log out / forget identity** `FR-id-logout`: The reviewer can log out, which forgets the app-stored identity — the stored secret key (for a local-key identity) or the stored bunker URI (for a bunker identity). After logout, published replies are unavailable until the reviewer logs in again or has an out-of-band identity configured. Logging out does not delete the public key history of already-published replies on Nostr (that is impossible by design); it only removes the app's local stored identity material.

### Bunker Login

The same screen offers a second identity form: a NIP-46 bunker connection URI. A bunker is a remote signer that holds the reviewer's secret key off-host and signs events on the reviewer's behalf over a Nostr relay. Logging in with a bunker lets a reviewer publish signed replies without their raw secret key ever being present on the review machine.

- **Login with bunker URI** `FR-id-bunker-login`: The app provides an in-app control on the identity screen for entering a NIP-46 bunker connection URI. On submit, the app parses the URI, connects to the remote signer named in the URI, obtains the reviewer's public key from the bunker, and adopts the bunker connection as the active identity for signing and attributing published replies. Signing under a bunker identity is delegated to the remote signer; the app holds no reviewer secret key in this mode. A malformed URI is rejected with a clear error and does not become the active identity. There is no "create a new bunker" action — a bunker is an external service the reviewer brings; the in-app bunker path is login-only.
- **Persist bunker identity across launches** `FR-id-bunker-persist`: A bunker identity adopted in-app persists across app launches by storing the bunker URI securely (never in plaintext on disk) and reconnecting to the remote signer on the next launch. The reviewer does not re-enter the URI per session. If the stored bunker URI cannot be read at launch, the app presents the login screen again. Reconnecting on launch re-runs the same connect handshake as the initial login; if the bunker is unreachable on a given launch, the identity is treated as unavailable for publishing for that session while read-only review and local commenting remain available (consistent with the existing bunker-failure behavior in `product/macos/shepherd-review.md` `FR-srm-bunker-sign-failure`).
- **Bunker connect failure during login** `FR-id-bunker-connect-failure`: If the bunker cannot be reached or refuses the connection during the in-app login (unreachable relay, bad secret, the signer rejects the handshake, or the request times out), the app does not adopt an identity, retains the entered URI so the reviewer can correct and retry, and shows a clear error naming the bunker as the cause. The reviewer remains on the identity screen. This is the login-time counterpart of the existing in-review bunker-sign-failure behavior (`FR-srm-bunker-sign-failure`); login failures are not silently dropped.

### Coexistence With Out-of-Band Identity

The app already loads a reviewer identity from environment variables and config files (see `FR-srm-identity-load` in `product/macos/shepherd-review.md`). The in-app screen coexists with that path rather than replacing it.

- **Out-of-band identity is honored** `FR-id-out-of-band-honored`: When a reviewer has configured an identity via the existing out-of-band paths (environment variable or config file), the app uses that identity and does not block on the in-app login screen. The in-app screen is the path for reviewers who have not configured an identity out of band. The precedence between out-of-band and in-app sources is an engineering decision; the product requirement is that an out-of-band identity, when present, is respected and the reviewer is not forced through the screen.
- **No silent override** `FR-id-no-silent-override`: Logging in or creating an identity in-app does not silently replace an identity the reviewer intentionally configured out of band without an indication. If both an in-app and an out-of-band identity are present, the app uses one and surfaces which one is active; it does not publish under an identity the reviewer did not expect.

### When the Screen Appears

The login screen is a gate for reviewers who have no usable identity, not a barrier for reviewers who do.

- **Screen shown when no identity** `FR-id-screen-when-no-identity`: The app presents the in-app identity screen when, at launch, no identity is available from any source (neither in-app stored nor out-of-band). A reviewer who has an identity available does not see the screen. The reviewer can dismiss the screen to proceed in a read-only / local-only mode (publishing unavailable), consistent with the existing behavior when no identity is configured.
- **Optional re-entry** `FR-id-optional-reentry`: A reviewer who is logged in can voluntarily open the identity screen again (e.g. to switch identities or create a new one) without being forced to log out first. This is the entry point for `FR-id-logout` and for replacing the active identity.

### Non-Functional Requirements

- **Secret key never in plaintext** `NFR-id-no-plaintext-key`: For a local-key identity, the secret key is never written to disk in plaintext, whether at rest (storage) or in transit (logs, crash reports, temporary files). Only the platform-appropriate secure storage mechanism holds the secret key, and only in memory otherwise. For a bunker identity, the stored bunker URI (which may carry a connection secret token) is held in the same secure storage and never written to disk in plaintext; the reviewer's secret key is never present on the host at all in this mode.
- **No key leaves the device** `NFR-id-key-stays-local`: For a local-key identity, the secret key never leaves the local machine. It is not transmitted to any remote service, relay, or third party. (Published Nostr events carry a signature, never the secret key.) For a bunker identity, the reviewer's secret key never lives on the local machine in the first place; only the bunker URI and a short-lived session keypair used solely for the encrypted connection to the remote signer (never as the reviewer's identity) are present locally.
- **Login is immediate (local-key)** `NFR-id-login-latency`: Adopting an entered or generated local-key identity (validation, public-key derivation, secure storage write) completes near-instantaneously; there is no network round-trip on the local-key path.
- **Bunker login is bounded** `NFR-id-bunker-connect-latency`: A bunker login involves a network round-trip (the handshake with the remote signer that obtains the reviewer's public key), so it is not instantaneous. The connect attempt is bounded by a timeout so the reviewer is not left waiting indefinitely on an unreachable bunker; a timed-out connect is treated as a connect failure (`FR-id-bunker-connect-failure`).
- **New key is cryptographically valid** `NFR-id-key-validity`: A generated secret key is a cryptographically valid Nostr secret key (a uniformly random scalar in the valid signing range) that derives to a valid Nostr public key, so the identity is immediately usable to sign events accepted by Nostr relays. The specific cryptographic curve and encoding are engineering decisions.

## Acceptance Criteria

- [ ] **Login with valid nsec** `AC-id-login-valid`: Given the identity screen is shown, when the reviewer pastes a valid `nsec1...` and submits, then the app adopts the identity, derives the matching public key, surfaces the active identity, and does not show the login screen on the next launch.
- [ ] **Login with invalid key rejected** `AC-id-login-invalid`: Given the identity screen is shown, when the reviewer enters a malformed string (not `nsec1...`, bad checksum, or not a 32-byte key) and submits, then the app shows a clear error naming the problem, does not adopt an identity, and remains on the screen.
- [ ] **Create new identity** `AC-id-create-new`: Given the identity screen is shown, when the reviewer chooses to create a new identity, then the app generates a fresh secret key, adopts it as the active identity, displays the resulting `nsec1...` for backup, and requires the reviewer to confirm they backed it up before the screen dismisses.
- [ ] **Created identity persists** `AC-id-create-persists`: Given the reviewer created an identity and confirmed backup, when the reviewer relaunches the app, then the identity is still active (no login screen) and the active identity indicator shows the same public key.
- [ ] **Created identity can publish** `AC-id-created-can-publish`: Given the reviewer created an identity in-app and is in a patch review, when the reviewer submits an inline comment, then the reply is signed under the created identity's public key and published, attributed to that identity.
- [ ] **Logout forgets identity** `AC-id-logout`: Given the reviewer is logged in via the in-app screen, when the reviewer logs out, then the app-stored secret key is removed, publishing becomes unavailable, and the next launch shows the login screen (assuming no out-of-band identity).
- [ ] **Out-of-band identity skips screen** `AC-id-out-of-band-skips`: Given the reviewer has configured an identity via an out-of-band path (environment variable or config file), when the reviewer launches the app, then the app uses that identity and does not show the in-app login screen.
- [ ] **Screen dismissable for read-only** `AC-id-dismiss-read-only`: Given the identity screen is shown, when the reviewer dismisses it without logging in, then the app proceeds in read-only / local-only mode and publishing is unavailable with a clear indication, consistent with existing no-identity behavior.
- [ ] **Active identity shown after login** `AC-id-active-shown`: Given the reviewer has logged in (in-app or out-of-band), when the reviewer views the app, then the active identity indicator shows the identity's display name or npub so the reviewer can confirm whose key their replies will be attributed to.
- [ ] **Secret key not in plaintext** `AC-id-no-plaintext`: Given the reviewer has logged in via the in-app screen, when the reviewer inspects the app's on-disk footprint (application support, preferences, logs), then the raw `nsec` / secret key bytes do not appear in plaintext anywhere on disk; only secure storage holds it.
- [ ] **Switch identity** `AC-id-switch`: Given the reviewer is logged in with identity A, when the reviewer opens the identity screen again and logs in with a different `nsec` B, then identity B becomes the active identity and the indicator reflects B; the previously stored key for A is replaced.
- [ ] **Login with valid bunker URI** `AC-id-bunker-login-valid`: Given the identity screen is shown, when the reviewer pastes a valid `bunker://` URI pointing at a reachable NIP-46 signer and submits, then the app connects to the signer, obtains the reviewer's public key, adopts the bunker identity, surfaces the active identity (with the bunker connection state visible), and does not show the login screen on the next launch (the stored URI reconnects).
- [ ] **Login with invalid bunker URI rejected** `AC-id-bunker-login-invalid-uri`: Given the identity screen is shown, when the reviewer enters a malformed bunker URI (not `bunker://`, missing relay, unparseable pubkey) and submits, then the app shows a clear error naming the problem, does not attempt a connection, does not adopt an identity, and remains on the screen.
- [ ] **Bunker connect failure during login** `AC-id-bunker-connect-failure`: Given the reviewer pastes a well-formed `bunker://` URI but the remote signer is unreachable or refuses the connection (bad secret, timeout), when the reviewer submits, then the app shows an error naming the bunker as the cause, retains the entered URI for correction, does not adopt an identity, and the reviewer can retry.
- [ ] **Bunker identity persists** `AC-id-bunker-persists`: Given the reviewer logged in with a bunker URI and then relaunches the app, when the app loads, then it reconnects to the bunker using the stored URI, the identity is active again (no login screen), and the indicator shows the bunker identity as connected.
- [ ] **Bunker identity can publish** `AC-id-bunker-can-publish`: Given the reviewer logged in with a bunker URI in-app and is in a patch review, when the reviewer submits an inline comment, then the reply is signed by the remote bunker under the reviewer's public key and published, attributed to that identity, without the reviewer's secret key ever being present on the host.
- [ ] **Bunker logout forgets URI** `AC-id-bunker-logout`: Given the reviewer is logged in via a bunker URI, when the reviewer logs out, then the stored bunker URI is removed, publishing becomes unavailable, and the next launch shows the login screen (assuming no out-of-band identity).
- [ ] **Bunker secret never on host** `AC-id-bunker-no-host-key`: Given the reviewer logged in with a bunker URI, when the reviewer inspects the app's on-disk footprint and memory, then the reviewer's Nostr secret key is not present anywhere (only the bunker URI in secure storage and an ephemeral control-channel keypair); the bunker URI (including any connection secret token) is not in plaintext on disk.

## Open Questions

1. **Out-of-band vs in-app precedence**: When both an out-of-band identity and an in-app stored identity exist, which wins? Options: (a) out-of-band wins (power users who set env vars are never surprised), (b) in-app wins (last action taken), (c) surface the conflict and let the reviewer choose. Recommendation deferred to engineering/design; the product requirement (`FR-id-no-silent-override`) is only that the reviewer is not surprised. See `product/macos/shepherd-review.md` `FR-srm-identity-load` for the existing out-of-band precedence list.

2. **Where the login screen sits in the launch flow**: Does the screen appear before the main review window opens (a gating onboarding window), or as a sheet/section within the review window that the reviewer reaches when they try to publish without an identity? This is a design decision; the product requirement (`FR-id-screen-when-no-identity`) is only that it appears when no identity is available.

3. **Confirm-backup enforcement strength**: When a new identity is created (`FR-id-show-new-nsec`), how strongly does the app enforce that the reviewer actually backed up the key? Options range from a light touch (display the key with a warning) to a heavier proof-of-possession gate (require the reviewer to demonstrate they have saved it) to a single explicit confirmation. A heavy gate adds friction; a too-light gate risks lost keys. The exact interaction is a design decision; the product requirement is only that the reviewer is warned this is their only chance to save the key and must acknowledge it before the screen dismisses.

4. **Logout vs replace semantics**: When a logged-in reviewer enters a new `nsec` via the re-entry path (`FR-id-optional-reentry`), is that an implicit logout-then-login, or a distinct "replace" action? The product requirement (`AC-id-switch`) only requires that the new identity becomes active and the old key is replaced; the exact interaction is a design decision.

## Dependencies

- **Reviewer identity model** (`product/macos/shepherd-review.md` `FR-srm-identity-load`, `FR-srm-identity-indicator`, `FR-sr-reviewer-identity`): The existing identity-load mechanism and the `ReviewerIdentity` model define what "an active identity" means and how it is surfaced. The in-app login feature produces an identity (local key or bunker) that feeds into the same model and indicator. Engineering will extend the identity-load precedence to include the in-app stored identity.
- **Event signing** (`product/macos/shepherd-review.md` `FR-srm-event-sign`): A local-key identity adopted via the in-app screen must be usable by the existing in-process signing path so published replies are signed under it.
- **Bunker signing** (`product/macos/shepherd-review.md` `FR-sr-bunker-signing`, `FR-srm-bunker-connect`, `FR-srm-bunker-sign-failure`): A bunker identity adopted via the in-app screen reuses the existing NIP-46 bunker control channel — connect handshake, `get_public_key`, and `sign_event` delegation — so published replies are signed by the remote signer under the reviewer's public key. The in-app bunker login is the login-time entry into the same bunker path the out-of-band configuration already feeds.
- **Platform secure storage**: Persisting the identity across launches (`FR-id-persistence`, `FR-id-bunker-persist`, `NFR-id-no-plaintext-key`) depends on a platform-appropriate secure storage facility that holds the secret key (local-key form) or the bunker URI (bunker form). The choice is an engineering decision documented in the platform engineering spec.
